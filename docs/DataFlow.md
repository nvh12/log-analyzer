## Hop 1 — `log.raw` (Simulation → log-processing)

### Message: `RawLog`

Published by Simulation as Pydantic `model_dump_json()`, consumed by log-processing as a Jackson-deserialized `RawLog` record. The consumer uses `setAlwaysConvertToInferredType(true)` to handle messages from Python (no `__TypeId__` header).

| Field | JSON key | Type | Nullable | Notes |
|---|---|---|---|---|
| `id` | `id` | string (UUID) | No | Auto-generated by Simulation; UUID v4 |
| `rawMessage` | `rawMessage` | string | No | CLF line (HTTP) or JSON string (FLOW) |
| `source` | `source` | enum `HTTP \| FLOW` | No | Determines parse path in log-processing |
| `receivedAt` | `receivedAt` | ISO-8601 datetime | Yes | Auto-set to UTC now by Simulation; log-processing defaults to `Instant.now()` if null |
| `headers` | `headers` | `dict[str, str]` | No | Defaults to `{}`. Only meaningful for HTTP track. |

**`rawMessage` for HTTP source** — Combined Log Format (CLF):
```
{ip} - - [{dd/MMM/yyyy:HH:mm:ss Z}] "{METHOD} {url} HTTP/1.1" {status} {size} "{referer}" "{ua}"
```
Referer and user-agent fields are optional (basic CLF vs. combined CLF). When absent, both fields parse as `null`.

**`rawMessage` for FLOW source** — JSON string:
```json
{
  "timestamp": float,
  "source_ip": "string",
  "dest_ip":   "string",
  "source_port": int,
  "dest_port":   int,
  "features": { "feature_name": float }
}
```
Any `NaN` or `Infinity` in `features` values is sanitized to `0.0` during parsing.

**Simulation-generated features note:** Simulation generates all 43 CICIDS2017 feature keys matching `flow_feature_cols.json`. When MinIO class stats (`flow/ddos/class_stats.json`, `flow/bruteforce/class_stats.json`) are available at startup, each feature is sampled from its per-class `[p25, p75]` interquartile range (`_flow_features_from_stats`). When MinIO is unreachable, a hardcoded fallback (`_flow_features_hardcoded`) generates plausible per-scenario ranges. Both paths produce vectors that are directly usable for ML inference. The `/simulate/replay` endpoint sends real CICIDS2017 CSV rows from MinIO verbatim for highest fidelity.

## Hop 4 — Redis sliding window (log-analysis internal)

The HTTP consumer adds each log to a Redis sorted set. The detection job reads it on schedule.

**Key:** `window:logs` (default; namespace-prefixed in deployment)
**Type:** Sorted Set
**Score:** `log.timestamp` (unix float)
**Member:** JSON-serialized `Log` dataclass via `dataclasses.asdict()`

The stored JSON matches the `Log` dataclass fields: `timestamp`, `ip`, `method`, `url`, `status_code`, `response_time_ms` (always 0.0), `response_size`, `query_string` (always `""`), `body` (always `null`), `headers` (always `{}`), `user_agent` (nullable), `referer` (nullable).

**Retention:** 60 seconds (configurable `WINDOW_SECONDS`). Entries older than the window are pruned on every 100th write and on every `get_window()` call.

## Hop 5 — Redis history (log-analysis internal)

Used by the traffic spike detection for EMA/Z-score/IQR baselines and seasonal baselines.

**Short-term history key:** `history:{key}` (list of floats, JSON array)
- Value: `[float, float, ...]` — request counts per window tick
- TTL: 7 days (configurable `HISTORY_TTL_SECONDS`)
- Truncated to last 60 entries on write

**Seasonal history key:** `history:{key}` (list of timed summaries, JSON array)
- Value: `[{"t": float, "m": float, "i": float}, ...]`
  - `t` — unix timestamp of the window bucket
  - `m` — median request count for that bucket
  - `i` — IQR of request count for that bucket
- Legacy V1 entries may use `{"t": ..., "c": count}` — treated as `(count, 0.0)`
- Max 1008 entries (6 weeks of hourly buckets)
- Deduplication: one entry per hour bucket (upsert by `floor(t / 3600) * 3600`)

Actually, re-reading the use cases: only anomalies are persisted AND published. `if result.anomaly: save(); publish()`. Non-anomaly results are dropped completely — not saved to DB, not published.

## Hop 10 — Redis reaction state (reaction → simulation reads)

The blocklist/rate-limit keys are set by Reaction service; Simulation's `AccessControlMiddleware` reads them to enforce access control on its own (simulated target) endpoints. `whitelist:ips` is the exception: it's owned by Simulation (written via its `/admin/whitelist` endpoints, exposed to the dashboard frontend) — Reaction only reads it, via an HTTP call to Simulation (`SimulationWhitelistClient`), not a direct Redis read, before deciding whether to block/rate-limit/alert.

### IP blocklist

| Key | Type | Value | TTL | Notes |
|---|---|---|---|---|
| `blocklist:ip:{ip}` | String | `"severity={SEVERITY};blocked_at={ISO_INSTANT}"` | Severity-based | Presence = blocked |
| `blocklist:ips` | Set | `{ip, ip, ...}` | None | Membership set for enumeration; stale entries self-clean via `isBlocked()` |
| `whitelist:ips` | Set | `{ip, ip, ...}` | None | Owned by Simulation; checked by Simulation's own middleware and (via HTTP) by Reaction before block/rate-limit; IPs here are never blocked |

### Rate limits

| Key | Type | Value | TTL | Notes |
|---|---|---|---|---|
| `ratelimit:ip:{ip}:limit` | String | requests-per-minute as string | Policy TTL | Presence = limited |
| `ratelimit:ip:{ip}` | String | `"0"` (counter) | 60 s (window TTL) | Request counter, resets each window |
| `ratelimit:ip:{ip}:window_end` | String | Unix timestamp as string | Policy TTL | Window anchor |

**Requests-per-minute by severity:**
- LOW: 30 / min
- MEDIUM: 10 / min
- HIGH: 3 / min
- CRITICAL: 1 / min

### DDoS and Brute force escalation counters

| Key | Type | Value | TTL | Notes |
|---|---|---|---|---|
| `ddos:attempts:{ip}` | String | integer counter | 600 s (10-min window) | Incremented atomically on each DDOS detection for this IP; expires after 10 minutes of inactivity |
| `brute:attempts:{ip}` | String | integer counter | 600 s (10-min window) | Incremented atomically on each BRUTE_FORCE detection for this IP; expires after 10 minutes of inactivity |

**Escalation logic (both DDOS and BRUTE_FORCE):** On first write the 600 s TTL is set (Lua atomic INCR+EXPIRE). If counter reaches `ESCALATION_THRESHOLD` (3), Reaction upgrades from `RATE_LIMIT` to `BLOCK` and calls `IpBlockService.block()`. Counter resets naturally when the 10-min TTL expires.

### Scale signals

| Key | Type | Value | TTL | Notes |
|---|---|---|---|---|
| `scale:state` | String | `"scaled_up"` | Severity-based | Presence = currently scaled |
| `scale:replicas` | String | integer as string | Severity-based | Target replica count |
| `config:scale:replicas:{SEVERITY}` | String | integer as string | None | Optional override; read before defaulting |

**Default replica targets by severity:** LOW=2, MEDIUM=3, HIGH=5, CRITICAL=8.

## Field-level anomalies and gaps

| Issue | Location | Impact |
|---|---|---|
| `response_time_ms` always 0.0 | `LogMessage` / `Log` in log-analysis | Feature extraction for web attack uses `response_size` but `response_time_ms` is unused noise |
| `body` always `None` | `LogMessage` / `Log` in log-analysis | `WebAttackInput.body` is always null; body-based attack patterns can never be detected |
| `headers` always `{}` | HTTP log path | log-processing copies `RawLog.headers` (which is `{}` from Simulation); actual HTTP headers are never captured |
| Non-anomalies completely dropped | All use cases | Neither DB row nor MQ event is produced for benign traffic — the `detection_results` table contains only confirmed attacks |
| `source_ip` null for TRAFFIC | `DetectionResult` / `reaction_logs` | TRAFFIC detections have no per-request IP; blocklist/rate-limit actions are not applicable; Reaction applies `SCALE_UP` instead |
| Dashboard drops `headers` column | `NormalizedHttpEntity` in Dashboard | The `headers` JSONB column in `log_processing.normalized_http` is invisible to Dashboard |
